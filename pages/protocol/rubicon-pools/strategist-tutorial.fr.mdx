## Stratège

Les stratèges ont la possibilité de placer une offre et/ou une demande en utilisant la liquidité BathToken à des fins de tenue de marché sur le carnet de commandes Rubicon. Ils utilisent des stratégies hors chaîne pour placer les transactions dans les limites du système de pools et gagner des récompenses.

### Points d'entrée et leviers du stratège

Le stratège du système Pools a trois fonctions clés qu'il peut utiliser pour faire du marketing sur Rubicon. Le stratège bénéficie de l'avantage de gagner une proportion de tous les remplissages qu'il crée tout en faisant du marché (voir `stratègeBootyClaim`) avec des fonds `bathToken`. Les trois fonctions principales sont les suivantes :

* `executeStrategy` - c'est la fonction principale qu'un stratège utilise pour placer des transactions dans le système en boucle fermée. Il permet de placer une offre et / ou une demande dans le carnet de commandes tout en enregistrant le stratège et les données clés afin qu'ils puissent être récompensés pour un comportement de tenue de marché positif en VAN
* `bathScrub` - cette fonction est la fonction de "nettoyage" obligatoire qui doit être effectuée sur les piscines Rubicon pour qu'elle fonctionne efficacement. Cette fonction analyse les commandes en cours sur un "BathPair" donné. Cela garantit qu'après "l'expiration du délai", toutes les commandes encore dans le carnet de commandes sont enregistrées pour le rendement et que la liquidité est renvoyée au pool. De plus, cette fonction doit être appelée pour réduire la quantité de commandes en cours pour une paire donnée afin de respecter `maxOutstandingPairCount`
* `removeLiquidity` - cette fonction permet à un stratège d'éviter les risques et de supprimer la liquidité du carnet de commandes qu'il a placé avec la liquidité des pools.

### Idées Clés

Les stratèges ont un point d'entrée central pour utiliser les fonds des utilisateurs et la création de marché sur le marché Rubicon : la fonction "executeStrategy" sur BathPair.sol. Le rôle d'un stratège est de trouver des offres exactes à faire avec les fonds des utilisateurs qui sont automatiquement, et exclusivement, placés dans le carnet de commandes Rubicon ; Les pools permettent de placer une offre ou une demande dans le carnet de commandes Rubicon.

Cette fonction a un certain nombre d'entrées __ qui correspondent directement à la fonction "offre" sur RubiconMarket.sol :

```
function executeStrategy(
        address targetStrategy, //address of the strategy to use - PairsTrade in v0
        uint256 askNumerator, // *Asset Amt* / Quote Amt
        uint256 askDenominator, // Asset Amt / *Quote Amt*
        uint256 bidNumerator, // *Quote Amt* / Asset Amt
        uint256 bidDenominator // Quote Amt / *Asset Amt*
    )
```

* `targetStrategy` - l'adresse du contrat exécutant une stratégie de tenue de marché avec les entrées données. Dans la version actuelle de Pools, le [_only approved strategy is the Pairs Trade_](/docs/protocol/rubicon-pools/bathpair).
* `askNumerator` - le numérateur exact ([pay\_am](/docs/protocol/rubicon-market/key-functions#offer)[t](https://www.youtube.com/watch?v=dQw4w9WgXcQ)) qui sera utilisé pour effectuer une transaction avec des fonds LP dans une quantité d'actifs (par exemple, WBTC ).
* `askDenominator` - le dénominateur exact ([buy\_amt](/docs/protocol/rubicon-market/key-functions#offer)) qui sera utilisé pour effectuer une transaction avec des fonds LP dans un montant d'actif de cotation (par exemple, USDC).
* `bidNumerator` - le numérateur exact ([pay\_am](/docs/protocol/rubicon-market/key-functions#offer)[t](https://www.youtube.com/watch?v=dQw4w9WgXcQ)) qui sera utilisé pour effectuer une transaction avec des fonds LP dans un montant d'actif de cotation (par exemple, USDC).
* `bidDenominator` - le dénominateur exact ([buy\_amt](/docs/protocol/rubicon-market/key-functions#offer)) qui sera utilisé pour faire cette offre dans un montant d'actif (par exemple WETH).

Le stratège doit déterminer hors chaîne les meilleures offres et demandes à faire en fonction des conditions du marché pour les fonds des utilisateurs afin de recevoir un paiement pour les remplissages réussis.
